## 核心主题概览

本视频围绕大模型推理中的两类缓存机制展开：**KV Cache** 与 **Prompt Cache / 提示词缓存**。二者本质上都在复用模型计算过程中产生的中间结果，以减少重复计算、降低延迟和成本，但作用范围不同：

- **KV Cache**：发生在**单次模型调用内部**，用于加速逐 token 生成。
- **Prompt Cache**：发生在**多次模型调用之间**，用于复用相同前缀提示词的 KV 结果。

所谓"缓存命中"，通常指后续请求的输入前缀与此前请求一致，因此可以直接复用已缓存的计算结果。

对 Agent、代码助手、长对话系统来说，Prompt Cache 的收益尤其明显，因为这些系统会反复调用模型，并且大量上下文是重复的。

> *批注：这期的重点不是"缓存"这个词本身，而是解释为什么大模型服务商会对缓存命中部分给出更低价格，以及为什么提示词设计会影响成本。*

---

## 核心概念

### 概念一：大模型生成文本的基本过程

大模型的本质是在不断预测下一个 token。

例如输入一句话后，模型并不是一次性生成完整回答，而是：

1. 根据当前输入预测第一个 token；
2. 把第一个 token 追加到输入末尾；
3. 再预测第二个 token；
4. 重复上述过程，直到输出结束标记 `EOS`。

在每次预测下一个 token 时，模型内部会基于输入 token 计算一些关键中间变量，其中最重要的是：

- `Q`：Query
- `K`：Key
- `V`：Value

> 每次计算下一个 token 时，需要用到前面 token 的 `K/V`，以及最后一个 token 的 `Q/K/V`。

---

### 概念二：什么是 KV Cache

在同一次模型调用中，随着模型逐 token 生成内容，前面 token 的 `K/V` 值会被反复用到。

关键点在于：同一个 token 的 `K/V` 在同一上下文中是固定的，只与当前 token、模型参数以及它之前的 token 有关。如果这些条件没变，`K/V` 就不会变，因此没有必要每次生成新 token 时都重新计算前面所有 token 的 `K/V`。

于是就有了 **KV Cache**：

1. 生成第一个 token 时，模型需要计算所有输入 token 的 `Q/K/V`；
2. 将所有 token 的 `K/V` 保存到缓存中；
3. 生成第二个 token 时，只需要计算新 token 的 `Q/K/V`；
4. 历史 token 的 `K/V` 直接从缓存中取出；
5. 后续生成过程持续复用已有 KV Cache。

这解释了一个常见现象：
- 模型输出第一个 token 往往更慢；
- 后续 token 输出速度会明显变快；
- 因为第一个 token 要处理完整输入，而后续 token 可以复用已有 `K/V`。

如果没有 KV Cache，每生成一个 token 都要重新计算整段上下文，生成速度会越来越慢。

> *KV Cache 解决的是"同一次回答里，别重复算已经算过的上下文"。它是大模型流式输出能够较快进行的基础优化之一。*

---

### 概念三：什么是 Prompt Cache / 提示词缓存

Prompt Cache 关注的是**多次模型调用之间**的缓存复用。

视频用"查询天气"的例子说明：用户问"明天天气怎么样？"，模型调用通常不是一次完成的，尤其当涉及工具调用时，会经历两次或多次模型调用。

- **第一次调用**：模型收到用户问题，判断需要调用天气工具，于是返回工具调用指令。
- **第二次调用**：后台完成工具调用后，需要再次调用模型，让模型根据工具结果生成自然语言回答。但模型本身是无状态的，所以第二次调用时不能只传工具结果，还必须把之前的对话记录一起传给模型。

其中，前一部分内容与第一次调用时高度相同。既然这些前缀在第一次调用时已经算过 `K/V`，那么第二次调用就可以直接复用。这就是 **Prompt Cache**。

> Prompt Cache 的定义：在多次模型调用中，如果后续调用与此前调用有相同的提示词前缀，那么这段前缀对应的 KV 结果可以被缓存并复用，无需重新计算。

---

## KV Cache 与 Prompt Cache 的区别

| 对比项 | KV Cache | Prompt Cache / 提示词缓存 |
|---|---|---|
| 作用范围 | 单次模型调用内部 | 多次模型调用之间 |
| 复用对象 | 当前调用中已生成/已处理 token 的 `K/V` | 之前调用中相同提示词前缀的 `K/V` |
| 解决问题 | 避免生成每个 token 时重复计算历史上下文 | 避免后续请求重复计算相同前缀 Prompt |
| 典型场景 | 一次回答逐 token 输出 | 多轮对话、工具调用、Agent 循环、代码助手 |
| 直接收益 | 提升生成速度 | 降低成本、降低延迟 |
| 命中条件 | 同一次调用中上下文自然延续 | 后续调用与历史调用拥有相同前缀 |

一句话总结：

> **KV Cache 是"这一轮生成里，前面算过的别再算"；Prompt Cache 是"下一轮调用里，前缀算过的别再算"。**

---

## 什么是缓存命中

"缓存命中"指的是：模型服务在处理新请求时，发现输入中的一段前缀已经在缓存中存在，于是直接复用缓存结果。

对于 Prompt Cache 来说，缓存命中的前提是：

- 后续请求的提示词前缀与之前请求一致；
- 前缀 token 没有发生变化；
- 服务端缓存仍然可用；
- 缓存隔离和索引正确。

因为 `K/V` 与前面的 token 强相关，只要前缀发生变化，后面 token 的 `K/V` 也会连带变化，缓存就无法继续复用。

> *缓存命中的本质不是"文本看起来差不多"，而是 token 级别的前缀必须一致。哪怕前面微小变动，也可能导致后续缓存无法复用。*

---

## 为什么缓存命中通常更便宜

缓存命中更便宜的核心原因是：服务商需要消耗的计算资源变少了。

尤其是长上下文、多轮对话、Agent 类应用中，大部分输入其实是历史记录，新增内容只占少量。

例如一个 Agent 执行任务时，用户可能只输入一句话，但后台会发生多次模型调用：模型理解任务 → 规划步骤 → 调用工具 → 工具返回结果 → 继续分析 → 再次调用工具 → 最终生成结果。每次调用都会带上前面的上下文，因此后续调用中绝大多数 token 都是重复历史。

只要缓存命中，服务商只需要为新增部分付出主要计算成本。在一次完整对话或 Agent 执行链路中，缓存命中比例可能非常高，视频中提到可达到 **98% 以上**。

> *这也是为什么有些 API 价格表会把输入 token 分成普通输入、缓存写入、缓存命中读取等不同计费项。真正的大头往往藏在长上下文和多次调用里。*

## 提高 Prompt Cache 命中率的提示词组织技巧

### 技巧一：静态提示词放前面，动态提示词放后面

由于 Prompt Cache 要求前缀一致，所以应尽量把不会变化的内容放在最前面。

适合放前面的静态内容：模型身份设定、行为规则、安全红线、工具使用规范、固定输出要求、长期不变的系统约束。

适合放后面的动态内容：当前时间、用户环境、用户语言偏好、项目状态、临时配置、当前任务上下文、个性化输出风格。

推荐结构：
```
固定系统规则
固定工具说明
固定安全约束
固定输出格式
---
动态环境信息
动态用户偏好
动态任务上下文
当前用户输入
```

### 技巧二：不要直接修改前缀，用追加声明保留前缀稳定

如果历史提示词中的某些信息发生变化，不要直接改动前面的系统提示词。例如原系统提示词中写的是"当前日期是 10 号"，到了凌晨变成 11 号时，如果直接把前缀里的 `10号` 改成 `11号`，那么前缀 token 发生变化，缓存可能失效。

更好的方式是保持原前缀不变，在后面追加一个变更声明：
```
<system-reminder>
当前日期已更新为 11 号。后续请以 11 号为准。
</system-reminder>
```

这样可以最大程度保持前缀稳定，让已有缓存继续发挥作用。

> *这类做法看起来有点"绕"，但对高频调用系统很实用。核心原则是：宁可在后面追加修正，也尽量不要改动前缀。*

## 对 Agent 开发的启发

如果正在开发 Agent 或大模型应用，可以重点关注以下设计原则：

1. **稳定系统提示词前缀**：不频繁改动 system prompt 的开头部分，把长期不变的规则集中放在最前面。
2. **变量信息后置**：时间、用户状态、任务环境等变量尽量放在后面，避免因为一个小变量变化导致整段缓存失效。
3. **减少无意义上下文扰动**：不要在前缀部分插入随机 ID、时间戳、动态日志等内容，这些内容会破坏缓存命中。
4. **明确区分静态与动态 Prompt**：静态部分用于缓存复用，动态部分用于表达当前任务差异。
5. **关注 API 服务商计费细项**：普通输入 token 价格、缓存写入价格、缓存命中读取价格、输出 token 价格、是否存在第三方加价或计费不透明。

## 缓存机制的风险：隐私与隔离问题

缓存虽然能显著提高效率、降低成本，但也存在风险。

最严重的问题是缓存隔离错误：如果缓存系统把 A 用户的缓存错误用于 B 用户的调用，模型就可能输出不该出现的信息，造成隐私泄露或上下文污染。

这类风险在以下场景中尤其敏感：多租户 API 平台、第三方中转服务、大规模 Agent 平台、企业知识库问答、代码助手、医疗/金融/法律等高隐私场景。

> *缓存不是单纯的性能优化，它已经进入安全工程范畴。只要涉及跨用户、跨会话、跨租户复用，就必须有严格隔离。*

## 本视频核心结论

1. **KV Cache 和 Prompt Cache 本质相近，都是复用模型计算中的 KV 中间结果。**
2. **KV Cache 作用于单次调用内部，负责加速逐 token 生成。**
3. **Prompt Cache 作用于多次调用之间，负责复用相同提示词前缀。**
4. **缓存命中意味着已有前缀的 KV 结果可以直接复用，因此计算成本更低。**
5. **Agent、代码助手、多轮对话场景中，Prompt Cache 收益很大，因为大量上下文会被反复发送。**
6. **提高缓存命中率的关键是保持前缀稳定：静态内容前置，动态内容后置。**
7. **如果必须更新信息，优先采用后置追加声明，而不是直接修改系统提示词前缀。**
8. **缓存管理不当可能导致上下文串线，甚至造成隐私泄露。**
9. **使用第三方 API 或中转商时，要特别关注缓存命中 token 的计费方式。**
